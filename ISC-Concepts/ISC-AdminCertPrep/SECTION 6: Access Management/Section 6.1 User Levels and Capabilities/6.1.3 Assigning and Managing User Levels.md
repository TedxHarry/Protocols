# 6.1.3 Assigning and Managing User Levels

**Learning Objective:** Learn how to assign, modify, and manage user access levels in SailPoint ISC.

---

## Understanding User Level Assignment

**Definition:** The process of granting ISC platform access to users and determining what they can do within ISC

**Key Concept:**
```
User Access Level Assignment:
  
  Step 1: User exists as identity in ISC
  Step 2: Grant ISC login access
  Step 3: Assign access level (Admin, Cert Admin, etc.)
  Step 4: User can log into ISC with assigned capabilities
```

**Important Distinction:**
```
Identity vs. ISC User:

Identity:
  - Person in ISC system
  - Has accounts in target systems (AD, Salesforce)
  - May or may not have ISC access
  - Example: 10,000 identities
  
ISC User:
  - Identity with ISC platform access
  - Can log into ISC
  - Has assigned access level
  - Example: 50 ISC users (out of 10,000 identities)
  
Not everyone needs ISC access!
  - Most employees don't need to log into ISC
  - Only IAM team, compliance, helpdesk need ISC access
```

**Exam Tip:** Identity exists in ISC ‚â† Can log into ISC. Must explicitly grant ISC access and assign access level.

## Prerequisites for Assigning Access Levels

### **What You Need Before Assigning**

**Prerequisite 1: Identity Must Exist**
```
User must be in ISC as an identity first

How identities get into ISC:
  ‚úì Aggregated from authoritative source (Workday, AD, HR system)
  ‚úì Manually created (rare, for special cases)
  
Example:
  John Smith exists in Workday
  ISC aggregates from Workday
  John Smith now exists as identity in ISC
  Identity ID: 2c91808a7a3c4f9e017a3c5b8f2e0123
```

**Prerequisite 2: Email Address Required**
```
Identity must have email address

Why:
  - ISC sends login invitation to email
  - Email used for notifications
  - Email used for password reset
  
If missing email:
  - Cannot grant ISC access
  - Must populate email first
  
Check:
  Admin ‚Üí Identities ‚Üí Search: John Smith
  Verify: email = john.smith@company.com ‚úì
```

**Prerequisite 3: Admin Access Level Required**
```
Only Admins can assign access levels

Who can grant ISC access:
  ‚úì Admin access level
  ‚úó Cert Admin (cannot grant access)
  ‚úó Helpdesk (cannot grant access)
  ‚úó Report Admin (cannot grant access)
  ‚úó User (cannot grant access)
  
Must be Admin to manage user access levels
```


## Revoking ISC Access

### **Removing ISC Access from User**

**When to Revoke:**
```
‚úì Employee terminates (leaver)
‚úì Role change (no longer needs ISC access)
‚úì Security incident (suspend access)
‚úì Temporary access expired
‚úì Admin reassignment (reduce admin count)
```

**Step 1: Navigate to User Access**
```
Login as: Admin
Navigate: Admin ‚Üí Admin ‚Üí User Access
```

**Step 2: Find User**
```
Search: john.smith@company.com

Current status:
  Name: John Smith
  Access Level: Helpdesk
  Status: Active
```

**Step 3: Revoke Access**
```
Click on: John Smith

Options:
  - Modify access level
  - Revoke access
  
Click: "Revoke Access" or "Remove User"

Confirmation prompt:
  "Are you sure you want to revoke ISC access for John Smith?"
  
  Warning:
    - User will no longer be able to log into ISC
    - All pending approvals will be reassigned
    - User will be notified (if configured)
    
Confirm: "Yes, Revoke Access"
```

**Step 4: Handle Reassignments (If Applicable)**
```
If user has pending tasks:

Pending Approvals:
  - Reassign to: Sarah Williams (new manager)
  
Active Certifications:
  - Reassign to: Mike Thompson (compliance team)
  
Scheduled Reports:
  - Reassign to: David Martinez (report admin)
  - Or disable reports
```

**Step 5: Verify Revocation**
```
User attempts to log in:
  Email: john.smith@company.com
  Password: [password]
  
Result:
  Error: "Access denied. You do not have permission to access this system."
  
Or:
  Account disabled
  Cannot authenticate
  
ISC Access: Revoked ‚úì
```

**Important Notes:**
```
Revoking ISC access:
  ‚úì Removes ISC platform access
  ‚úì User cannot log into ISC
  
Does NOT affect:
  ‚úó Identity in ISC (still exists)
  ‚úó Accounts in target systems (AD, Salesforce still active)
  ‚úó Access in other systems
  
To remove all access:
  1. Revoke ISC access (platform)
  2. Trigger leaver workflow (de-provision from targets)
```

```
  HR Team ‚Üí Owns Workday source
```

```
  ‚òë Configure source settings
```

## Access Level Best Practices

### **Security and Governance**

**Principle of Least Privilege:**
```
‚úì Grant minimum access needed
‚úì Default to User level for most
‚úì Helpdesk only for support staff
‚úì Admin only for IAM team (2-5 people)

Example Distribution (10,000 employees):
  Admin: 3 people (0.03%)
  Cert Admin: 5 people (0.05%)
  Helpdesk: 20 people (0.2%)
  Report Admin: 5 people (0.05%)
  User: 0 people (most don't need ISC access)
  No ISC Access: 9,967 people (99.67%)
  
Only 33 people need ISC access!
```

**Separation of Duties:**
```
‚úì Separate Admin and Cert Admin roles
‚úì Compliance team = Cert Admin (not Admin)
‚úì IAM team = Admin (not Cert Admin)

Why:
  - Prevent conflict of interest
  - Compliance team certifies access
  - IAM team configures system
  - Should be different people
  
Example:
  IAM Team (Admin):
    - Sarah Johnson (IAM Lead)
    - Mike Chen (IAM Engineer)
    
  Compliance Team (Cert Admin):
    - Lisa Anderson (Compliance Manager)
    - Tom Wilson (Internal Audit)
    
  No overlap = Good separation
```

**Regular Access Reviews:**
```
‚úì Review ISC access quarterly
‚úì Verify users still need access
‚úì Remove unnecessary access
‚úì Validate access levels appropriate

Quarterly Review Process:
  1. Export all ISC users
  2. Review each user:
     - Still in role requiring ISC access?
     - Access level still appropriate?
     - Still employed?
     
  3. Actions:
     - Revoke: Terminated employees
     - Revoke: Role changed (no longer needs ISC)
     - Modify: Access level change needed
     - Maintain: Access still appropriate
     
  4. Document review
  5. Report to management
```

**Documentation:**
```
‚úì Document why each user has ISC access
‚úì Document access level rationale
‚úì Track access changes

Example Documentation:
  User: sarah.williams@company.com
  Access Level: Helpdesk
  Granted: 2024-01-15
  Granted By: admin@company.com
  Reason: IT Support role, unlock accounts
  Business Justification: Support team member
  Last Reviewed: 2024-02-01
  Status: Active
```

## Troubleshooting Access Level Issues

### **Common Issues and Solutions**

**Issue 1: User Cannot Log Into ISC**

**Symptoms:**
```
User tries to log in:
  Email: john.smith@company.com
  Password: [password]
  
Result:
  Error: "Invalid credentials" or "Access denied"
```

**Diagnosis:**
```
Check 1: Does identity exist?
  Admin ‚Üí Identities ‚Üí Search: john.smith
  
  If NOT found:
    - Identity doesn't exist in ISC
    - Need to aggregate from source first
    
  If found:
    - Identity exists ‚úì
    - Continue to Check 2

Check 2: Has ISC access been granted?
  Admin ‚Üí Admin ‚Üí User Access
  Search: john.smith@company.com
  
  If NOT in list:
    - No ISC access granted
    - User is identity but not ISC user
    - Solution: Grant ISC access
    
  If in list:
    - ISC access exists ‚úì
    - Continue to Check 3

Check 3: Is access active?
  View user: john.smith@company.com
  Status: Active or Inactive?
  
  If Inactive:
    - Access was revoked or suspended
    - Solution: Reactivate access
    
  If Active:
    - Access should work ‚úì
    - Continue to Check 4

Check 4: Has user activated account?
  Check: Invitation status
  
  If "Invitation Pending":
    - User hasn't clicked invitation link
    - User hasn't set password
    - Solution: Resend invitation
    
  If "Active":
    - Account activated ‚úì
    - Password should work
    - May be password issue
```

**Solution:**
```
Most common: ISC access not granted

Fix:
  1. Admin ‚Üí Admin ‚Üí User Access
  2. Add User: john.smith@company.com
  3. Assign access level: Helpdesk
  4. ‚òë Send welcome email
  5. Save
  
  User receives invitation
  Sets password
  Can now log in ‚úì
```

```

```

```
    - IT Support Lead (only needs user support)

    - Support Engineer (only needs user support)
```

## Section Summary

**Core Concepts for Certification:**

‚úÖ **Prerequisites for Assigning Access:**
- Identity must exist in ISC first
- Identity must have email address
- Only Admin can assign access levels
- User receives invitation email to activate

‚úÖ **Granting Access Process:**
1. Verify identity exists
2. Admin ‚Üí Admin ‚Üí User Access
3. Add User / Grant Access
4. Select identity and assign access level
5. Send invitation email
6. User sets password and activates
7. User can log in with assigned capabilities

‚úÖ **Modifying Access:**
- Find user in User Access management
- Change access level (upgrade or downgrade)
- Save changes (takes effect immediately)
- User has new capabilities on next login
- Document changes for audit

‚úÖ **Revoking Access:**
- Check for pending tasks (approvals, certifications, reports)
- Reassign pending tasks to other users
- Revoke ISC access
- User cannot log into ISC
- Identity and target system accounts unaffected

‚úÖ **Best Practices:**
- Principle of least privilege (minimum access needed)
- Most users don't need ISC access
- Separate Admin and Cert Admin (separation of duties)
- Regular access reviews (quarterly)
- Document all access changes
- Limit Admin access to 2-5 people

‚úÖ **Common Issues:**
- User can't log in ‚Üí ISC access not granted (grant access)
- Wrong capabilities ‚Üí Wrong access level (modify level)
- Too many Admins ‚Üí Security risk (rightsize access)
