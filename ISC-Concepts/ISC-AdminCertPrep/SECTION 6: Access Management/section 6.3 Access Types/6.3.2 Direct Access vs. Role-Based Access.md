# 6.3.2 Direct Access vs. Role-Based Access

**Learning Objective:** Understand the differences between direct access and role-based access, and when to use each approach.

---

## Direct Access Explained

### **What is Direct Access?**

**Definition:** Individual entitlements assigned directly to a user's account, independent of any role

**Simple Explanation:**
```
Direct Access = Granting specific entitlements one-by-one

Example:
  User: John Smith
  Direct entitlements:
    - VPN-Users (granted individually)
    - Sales-Team (granted individually)
    - Project-Alpha (granted individually)
    - Finance-Reports (granted individually)
    
  Each entitlement granted separately
  No role involved
  Individual assignments
```

**Visual Representation:**
```
Identity: John Smith
  ↓
  ├─ VPN-Users (direct)
  ├─ Sales-Team (direct)
  ├─ Project-Alpha (direct)
  └─ Finance-Reports (direct)
  
Each arrow = separate assignment
No grouping or bundling
```

**Exam Tip:** Direct access = individual entitlements granted separately. No role, no bundle, just specific entitlement to specific person.

---

### **Characteristics of Direct Access**

**Key Attributes:**
```
✓ Granular (one entitlement at a time)
✓ Explicit (specifically granted)
✓ Flexible (any combination possible)
✓ Independent (not tied to role)
✓ Visible as direct assignment
```

**Assignment Methods:**
```
How direct access is granted:

1. Self-Service Request:
   User requests → Approval → Granted
   
2. Manager Request:
   Manager requests for user → Approval → Granted
   
3. Admin Provisioning:
   Admin directly assigns → Granted
   
4. Automated (non-role):
   Rule-based assignment (not via role)
   Example: IF project="Alpha" THEN grant Project-Alpha
```

### **Direct Access Examples**

**Example 1: Project-Based Access**
```
Scenario:
  Employee: Sarah Chen (IT Developer)

  Entitlement: Phoenix-Project-Team

  - Temporary project assignment
  - Not all IT developers need it
  - Specific to this project

Result:
  Sarah has Phoenix-Project-Team (direct)
  Plus her normal IT Developer role entitlements
```

  Granted to: Mike Johnson
  Duration: 30 days
  Reason: "Q4 campaign requires sales data"
  
Not in Marketing role:
  - Marketing role doesn't include sales reports
  - Exception for specific need
  - Temporary access
  - Not appropriate for role
  
Result:
  Mike has Sales-Reports-Access (direct, temporary)
  Plus his normal Marketing role entitlements
```

**Example 3: Individual Tool Access**
```
Scenario:

Result:
  Lisa has Excel-Advanced-Analytics-License (direct)
  Plus her Finance Analyst role entitlements
```

### **When to Use Direct Access**

**Appropriate Scenarios:**
```
✓ Temporary/project-based access
✓ Exception cases (not standard for role)
✓ Individual tool/license assignments
✓ One-off requests
✓ Testing/troubleshooting
✓ Access that expires
✓ Unique individual needs
```

**Examples:**

**1. Temporary Project:**
```
Project: Data migration (3 months)
Team: 5 people from different departments

Solution: Direct access
  Grant: Migration-Project-Access (directly)
  To: 5 specific individuals
  Duration: 3 months
  
After project:
  Revoke from all 5 people
  Access no longer needed
  
Why not role:
  - Temporary need
  - Cross-functional team
  - Specific project
  - Time-limited
```

**2. Individual License:**
```
Tool: Specialized software ($500/license)
Need: Only 1 person in department

Solution: Direct access
  Grant: Adobe-Creative-Cloud license
  To: Graphic designer (only person who needs it)
  Duration: Permanent (while in role)
  
Why not role:
  - Only 1 person needs it
  - Expensive license
  - Not standard for department
  - Individual requirement
```

**3. Exception Request:**
```
Requester: Executive assistant
Need: Access to executive's calendar
Standard: Admins don't normally have this access

Solution: Direct access
  Grant: Executive-Calendar-Access
  To: Executive assistant
  Duration: While supporting this executive
  
Why not role:
  - Exception case
  - Not all assistants need this executive's calendar
  - Specific relationship
  - Individual need
```

---

### **Advantages of Direct Access**

**Flexibility:**
```
✓ Can grant any entitlement to anyone
✓ No role constraints
✓ Custom combinations possible
✓ Adapts to unique needs

Example:
  User needs entitlements A, B, C, F, G, Q
  No role matches this combination
  Grant directly ✓
```

**Simplicity (for small scale):**
```
✓ No role design needed
✓ No role maintenance
✓ Direct and straightforward
✓ Easy to understand

Example:
  Small company (20 employees)
  Each person unique mix of access
  Direct access is simpler than complex role model
```

**Granular Control:**
```
✓ Precise access assignments
✓ Individual-level decisions
✓ Fine-tuned access control
✓ Specific approvals

Example:
  Need to grant VPN access to 1 person
  Don't want to create role for 1 person
  Grant directly ✓
```

---

### **Disadvantages of Direct Access**

**Access Sprawl:**
```
Problem:
  Every user has different mix of entitlements
  No standardization
  Impossible to manage at scale
  
Example:
  Sales Team (50 people):
    Rep 1: VPN, Salesforce, Sales-Team, Marketing-Tools, Reports
    Rep 2: VPN, Salesforce, Sales-Team, HR-Portal, Finance
    Rep 3: VPN, Salesforce, IT-Tools, Project-A
    ... all different!
    
  Why different? They're all sales reps!
  Should have same access (role-based would fix this)
```

**Complex Certifications:**
```
Problem:
  Must certify each entitlement individually
  Many decisions per person
  Time-consuming and error-prone
  
Example:
  Manager certifying 10 team members:
    Each person has 8 direct entitlements
    10 people × 8 entitlements = 80 decisions
    
  With roles:
    Each person has 1 role
    10 people × 1 role = 10 decisions
    
  88% more work with direct access!
```

**Onboarding Complexity:**
```
Problem:
  Must manually grant each entitlement
  Easy to miss something
  Inconsistent onboarding
  
Example:
  New sales hire needs:
    1. Request VPN-Users
    2. Request Salesforce-Access
    3. Request Sales-Team
    4. Request CRM-Tools
    5. Request Sales-Reports
    6. Request Mobile-Access
    7. Request Sales-Shared-Drive
    8. Request Email-Sales-Alias
    
  8 separate requests!
  User confusion, manager approval fatigue
  
  With role:
    Assign "Sales Representative" role → All 8 granted automatically
```

**No Standardization:**
```
Problem:
  People in same job have different access
  No consistency
  Hard to audit
  
Example:
  Sales Rep A has: VPN, Salesforce, Sales-Team
  Sales Rep B has: VPN, Salesforce, Sales-Team, Reports, Marketing
  Sales Rep C has: VPN, Salesforce
  
  Why different? Same job!
  Audit asks: "What should sales reps have?"
  Answer: "It varies..." ✗
  
  With role:
    All sales reps have "Sales Representative" role
    Identical access ✓
    Easy to audit ✓
```

**Difficult to Maintain:**
```
Problem:
  Access changes require updating each person individually
  Time-consuming
  Error-prone
  
Example:
  New tool required for all sales reps: Sales-Analytics
  
  Direct access approach:
    Manually grant to: Rep 1, Rep 2, Rep 3, ... Rep 50
    50 individual grants
    Easy to miss someone
    
  Role-based approach:
    Add to "Sales Representative" role
    All 50 reps get it automatically ✓
    One change, everyone updated ✓
```

## Role-Based Access Explained

### **What is Role-Based Access?**

**Definition:** Access granted through assignment to a role that bundles multiple entitlements representing a job function

**Simple Explanation:**
```
Role-Based Access = Assign user to role → Get all entitlements in role

Example:
  User: John Smith
  Role: Sales Representative
  
  Role contains:
    - VPN-Users
    - Sales-Team
    - Salesforce
    - Sales-Reports
    - CRM-Tools
    - Mobile-Access
    
  Assign John to role → John gets all 6 entitlements automatically
```

**Visual Representation:**
```
Identity: John Smith
  ↓
Role: Sales Representative
  ↓
  ├─ VPN-Users
  ├─ Sales-Team
  ├─ Salesforce
  ├─ Sales-Reports
  ├─ CRM-Tools
  └─ Mobile-Access
  
One assignment (role) → Multiple entitlements
Bundled and managed as unit
```

**Exam Tip:** Role-based access = assign role once, get all role's entitlements automatically. Roles represent job functions.

---

### **Characteristics of Role-Based Access**

**Key Attributes:**
```
✓ Bundled (multiple entitlements in role)
✓ Job-based (represents job function)
✓ Consistent (everyone in role gets same access)
✓ Maintainable (update role → everyone updated)
✓ Scalable (works for thousands of users)
```

**Role Structure:**
```
Role Definition:
  Name: Sales Representative
  Description: "Standard access for sales team members"
  
  Entitlements (6):
    1. VPN-Users (AD)
    2. Sales-Team (AD)
    3. Salesforce Sales Cloud User (Salesforce)
    4. Sales-Reports-Access (Reporting)
    5. CRM-Tools-Access (CRM)
    6. Mobile-Access (Mobile)
    
  Assignment Criteria:
    IF department = "Sales" AND jobTitle CONTAINS "Representative"
    
  Approval:
    Manager approval required
    
  Certification:
    Quarterly by sales managers
```

### **When to Use Role-Based Access**

**Appropriate Scenarios:**
```
✓ Standard job functions
✓ Multiple people in same role
✓ Consistent access requirements
✓ Long-term access needs
✓ Cross-system access
✓ Scalable environments
✓ Mature IAM programs
```

**Decision Criteria:**
```
Use Role-Based Access When:

1. Multiple people have same job function
   Example: 50 sales reps (all need same access)
   
2. Access is permanent (not temporary)
   Example: Job-based access (not project-based)
   
3. Access spans multiple systems
   Example: AD + Salesforce + CRM + Reports
   
4. Consistency is important
   Example: All analysts should have identical access
   
5. Frequent onboarding/offboarding
   Example: Sales team with high turnover
   
6. Need simplified certifications
   Example: Certify role membership, not individual entitlements
```

---

### **Advantages of Role-Based Access**

**Consistency:**
```
Benefit:
  Everyone in role gets identical access
  No variance, no confusion
  
Example:
  Sales Representative role:
    Rep 1: Gets entitlements A, B, C, D, E, F
    Rep 2: Gets entitlements A, B, C, D, E, F
    Rep 3: Gets entitlements A, B, C, D, E, F
    ... all identical ✓
    
  Audit question: "What access do sales reps have?"
  Answer: "Sales Representative role entitlements" ✓
  Clear and consistent
```

**Simplified Onboarding:**
```
Benefit:
  One role assignment → All access granted
  Fast, complete, consistent
  
Example:
  New hire: Day 1
  
  Direct access approach:
    Submit 8 access requests
    Wait for 8 approvals
    Receive 8 grants
    Time: 1-3 days
    Risk: Miss something
    
  Role-based approach:
    Assign "Sales Representative" role
    All 8 entitlements granted automatically
    Time: Minutes
    Risk: Nothing (complete and consistent)
```

**Simplified Offboarding:**
```
Benefit:
  Remove one role → All access revoked
  Complete, fast, no leftovers
  
Example:
  Termination: Final day
  
  Direct access approach:
    Find all entitlements (hope you find them all)
    Revoke individually: A, B, C, D, E, F, G, H
    Time: 30 minutes
    Risk: Miss something (orphaned access)
    
  Role-based approach:
    Remove "Sales Representative" role
    All entitlements revoked automatically
    Time: 1 minute
    Risk: Nothing (complete revocation)
```

**Simplified Certifications:**
```
Benefit:
  Certify role membership, not individual entitlements
  Fewer decisions, faster process
  
Example:
  Manager certifying 50 sales reps:
  
  Direct access (8 entitlements each):
    50 people × 8 entitlements = 400 decisions
    Questions: Does person need A? B? C? D? E? F? G? H?
    Time: Hours
    
  Role-based (1 role each):
    50 people × 1 role = 50 decisions
    Question: Is person still Sales Representative?
    Time: 30 minutes
    
  88% time savings ✓
  Much simpler ✓
```

**Easy Maintenance:**
```
Benefit:
  Update role → All members updated automatically
  One change affects everyone
  
Example:
  New requirement: All sales reps need Sales-Analytics tool
  
  Direct access approach:
    Grant to: Rep 1, Rep 2, Rep 3, ... Rep 50
    50 individual grants
    Time: 1-2 hours
    Risk: Miss someone
    
  Role-based approach:
    Add Sales-Analytics to "Sales Representative" role
    All 50 reps get it automatically
    Time: 2 minutes
    Risk: Nothing (everyone updated)
```

**Scalability:**
```
Benefit:
  Works for 10 users or 10,000 users
  No additional complexity
  
Example:
  Company grows: 50 → 500 sales reps
  
  Direct access:
    Complexity increases 10x
    500 people to manage individually
    Unmanageable
    
  Role-based:
    Complexity unchanged
    Still just "Sales Representative" role
    500 people managed as easily as 50 ✓
```

**Auditability:**
```
Benefit:
  Clear role definitions
  Easy to understand and audit
  Compliance-friendly
  
Example:
  Auditor: "What access do finance analysts have?"
  
  Direct access answer:
    "Well, it varies... let me check each person..."
    Pull reports for all 25 analysts
    Different access for each
    Hard to explain
    
  Role-based answer:
    "Finance Analyst role provides these 8 entitlements"
    Show role definition
    All 25 have same access
    Clear and defensible ✓
```

### **Disadvantages of Role-Based Access**

**Initial Design Effort:**
```
Challenge:
  Must design role model upfront
  Define roles and entitlements
  Time investment initially
  
Effort:
  - Identify job functions
  - Define roles
  - Map entitlements to roles
  - Create role hierarchy
  - Document roles
  
Time: Weeks to months (initially)

Trade-off:
  Upfront effort, long-term benefit
  Worth it for mature programs
```

**Role Explosion:**
```
Challenge:
  Too many roles (over-specialization)
  Hard to manage
  
Example (Bad):
  - Sales-Rep-East-Coast
  - Sales-Rep-West-Coast
  - Sales-Rep-Central
  - Sales-Rep-Senior
  - Sales-Rep-Junior
  - Sales-Rep-Enterprise
  - Sales-Rep-SMB
  ... 50 different sales roles!
  
Better:
  - Sales Representative (one role)
  - Use direct access for exceptions
  
Guideline:
  Aim for 20-50 roles (not 200-500)
```

**Less Flexibility:**
```
Challenge:
  Roles are predefined
  May not fit every individual perfectly
  
Example:
  Sales Rep needs 95% of Sales Representative role
  But doesn't need VPN (works in office only)
  
Options:
  A) Grant full role (includes VPN they don't need)
  B) Use direct access instead (loses role benefits)
  C) Create new role without VPN (role explosion)
  
Usually:
  Grant full role, accept small amount of unnecessary access
  Better than losing role benefits
```

**Maintenance Overhead:**
```
Challenge:
  Roles need ongoing maintenance
  Update as business changes
  
Examples:
  - New tool adopted → Add to relevant roles
  - Tool decommissioned → Remove from roles
  - Job functions change → Update role definition
  - Reorganization → Redesign roles
  
Ongoing effort required:
  Quarterly role reviews
  Annual role redesign (if needed)
```

## When to Use Which Approach

### **Use Direct Access When:**

**Criteria:**
```
✓ Temporary access (projects, limited duration)
✓ Exception cases (doesn't fit role)
✓ Individual/unique needs (only 1 person)
✓ Testing/troubleshooting
✓ Very small organization (<20 people)
✓ One-off requests
✓ Access expires soon
```

**Examples:**
```
1. Temporary Project:
   5 people need Project-Phoenix access for 3 months
   → Direct access (time-limited, temporary team)
   
2. Individual License:
   1 person needs expensive specialized software
   → Direct access (only person who needs it)
   
3. Exception:
   Marketing person needs finance reports (unusual)
   → Direct access (exception, not in role)
   
4. Small Company:
   Company has 15 employees, all unique roles
   → Direct access simpler than role design
```

---

### **Use Role-Based Access When:**

**Criteria:**
```
✓ Multiple people same job function (10+ people)
✓ Permanent access (not temporary)
✓ Standard job functions (repeatable)
✓ Cross-system access (AD + apps + tools)
✓ Need consistency
✓ Frequent onboarding/offboarding
✓ Scalability required
✓ Mature IAM program
```

**Examples:**
```
1. Sales Team:

## Migration from Direct to Role-Based

### **Migration Process**

**Step 1: Analyze Current State**
```
Review current access:
  - Identify common patterns
  - Group similar users
  - Find standard access sets
  
Example:
  Analysis of 50 sales reps:
    Common entitlements (all 50 have):
      - VPN-Users
      - Sales-Team
      - Salesforce
      - Sales-Reports
      - CRM-Tools
      
    Variable entitlements (some have):
      - Marketing-Tools (10 people)
      - Finance-Reports (5 people)
      - Project-specific (varies)
```

**Step 2: Design Roles**
```
Create role for common access:
  
Role: "Sales Representative"
  Include common entitlements:
    - VPN-Users
    - Sales-Team
    - Salesforce
    - Sales-Reports
    - CRM-Tools
    
  Exclude variable entitlements:
    Leave as direct access (exceptions)
```

**Step 3: Pilot**
```
Test with small group:
  - Select 5-10 sales reps
  - Assign "Sales Representative" role
  - Verify they still have all access
  - Validate no disruption
  - Gather feedback
```

**Step 4: Migrate in Phases**
```
Phase 1: 10 sales reps
  Assign role, validate
  
Phase 2: 20 more sales reps
  Assign role, validate
  
Phase 3: Remaining 20 sales reps
  Assign role, validate
  
Gradual rollout reduces risk
```

**Step 5: Clean Up**
```
After role assignment:
  - Review direct entitlements
  - Remove those now covered by role
  - Keep only true exceptions
  
Example:
  Before:
    5 direct entitlements
    
  After:
    Role covers 5 entitlements
    Direct: Only 0-2 exceptions remain
```
