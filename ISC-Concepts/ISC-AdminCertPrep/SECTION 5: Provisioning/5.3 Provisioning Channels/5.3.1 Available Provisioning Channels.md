# 5.3.1 Available Provisioning Channels

**Learning Objective:** Understand the different provisioning channels available in ISC and how they work.

---

## What is a Provisioning Channel?

**Definition:** The method or pathway ISC uses to deliver provisioning commands to target systems

**Simple concept:**
```
Provisioning Channel = How ISC communicates provisioning to the target

Example:
  Channel: Direct API
  ISC → Salesforce API → Account Created
```

**Key point:**
```
Different sources use different channels based on:
  - Where the source is (cloud vs. on-premises)
  - What protocols it supports (API, LDAP, etc.)
  - Security requirements
  - Integration capabilities
```

**Exam Tip:** Provisioning channel = the communication method ISC uses to provision. Channel choice depends on source type and location.

---

## Available Provisioning Channels

### **Overview of Channels**

| Channel | Description | Use Case | Example Sources |
|---------|-------------|----------|-----------------|
| **Direct API** | ISC calls source API directly | Cloud SaaS apps | Salesforce, Workday, ServiceNow |
| **Virtual Appliance (VA)** | ISC → VA → Source | On-premises systems | Active Directory, Oracle DB, LDAP |
| **Web Services** | ISC calls REST/SOAP APIs | Modern applications | Custom apps, cloud services |
| **SCIM** | Standard provisioning protocol | Cloud apps with SCIM | Slack, Zoom, GitHub |
| **Cloud Gateway** | Cloud connector service | Hybrid scenarios | Azure AD, Google Workspace |

## Channel 1: Direct API (Cloud Connector)

### **How It Works**

**Architecture:**
```
ISC (Cloud) → Internet → Target API (Cloud) → Account Created

No intermediaries
Direct connection
Real-time provisioning
```

**Visual flow:**
```
┌─────────────┐
│     ISC     │
│   (Cloud)   │
└──────┬──────┘
       │ HTTPS/API Call
       │
       ↓
┌─────────────┐
│  Salesforce │
│     API     │
└──────┬──────┘
       │
       ↓
Account Created
```

### **Characteristics**

**Advantages:**
```
✓ Fast (no intermediary)
✓ Real-time provisioning
✓ Simple architecture (no VA needed)
✓ Automatic updates (SailPoint manages connectors)
✓ High availability (cloud-to-cloud)
✓ Scalable
```

**Disadvantages:**
```
✗ Requires internet connectivity
✗ Both systems must be cloud-based
✗ API rate limits may apply
✗ Dependent on target API availability
```

### **Authentication Methods**

**OAuth 2.0:**
```
Most common for modern cloud apps

Flow:
  1. ISC requests OAuth token
  2. Target responds with access token
  3. ISC uses token for API calls
  4. Token refreshes automatically

Example - Salesforce:
  ISC authenticates via OAuth
  Receives access token
  Uses token for all provisioning API calls
```

**API Key:**
```
Static authentication

Flow:
  1. Admin generates API key in target system
  2. Admin enters API key in ISC
  3. ISC includes key in all API calls

Example - ServiceNow:
  Admin creates integration user
  Generates API key
  ISC uses key for provisioning
```

**Basic Auth:**
```
Username and password

Flow:
  ISC sends credentials with each request
  Less common for modern apps
  
Example - Legacy systems:
  Username: integration@company.com
  Password: [stored in ISC]
```

## Channel 2: Virtual Appliance (VA)

### **How It Works**

**Architecture:**
```
ISC (Cloud) → VA (On-Prem) → Target System (On-Prem) → Account Created

VA acts as intermediary
Executes provisioning locally
Reports results to ISC
```

**Visual flow:**
```
┌─────────────┐
│     ISC     │
│   (Cloud)   │
└──────┬──────┘
       │ HTTPS (Outbound)
       │ Provisioning Command
       ↓
┌─────────────┐
│     VA      │
│  (On-Prem)  │
└──────┬──────┘
       │ LDAP/PowerShell/JDBC
       │ Local Network
       ↓
┌─────────────┐
│  Active     │
│  Directory  │
│  (On-Prem)  │
└─────────────┘
Account Created
```

### **Characteristics**

**Advantages:**
```
✓ Reaches on-premises systems
✓ Secure (no inbound connections required)
✓ Uses existing network/security
✓ Can access multiple local sources
✓ Supports legacy protocols (LDAP, JDBC, etc.)
```

**Disadvantages:**
```
✗ Requires VA infrastructure
✗ VA maintenance needed (updates, monitoring)
✗ Additional point of failure
✗ Slightly slower (extra hop)
✗ Requires on-premises resources
```

### **VA Communication**

**ISC to VA:**
```
Protocol: HTTPS (outbound from VA)
Direction: VA initiates connection (polls ISC)
Frequency: Continuous (websocket-style)

Process:
  1. VA maintains connection to ISC
  2. ISC queues provisioning commands
  3. VA polls/receives commands
  4. VA executes locally
  5. VA reports results to ISC
  
Security:
  ✓ Certificate-based authentication
  ✓ Encrypted communication (TLS 1.2+)
  ✓ No inbound firewall rules needed
```

**VA to Target System:**
```
Protocols vary by connector:
  - LDAP/LDAPS (Active Directory, OpenLDAP)
  - JDBC (Oracle, SQL Server, MySQL)
  - PowerShell (Active Directory, Exchange)
  - SSH (Unix/Linux systems)
  - HTTP/HTTPS (Web services)

Process:
  1. VA receives command from ISC
  2. VA connects to local target
  3. VA executes provisioning operation
  4. Target responds with success/failure
  5. VA reports to ISC
```

### **VA Provisioning Examples**

**Example 1: Active Directory**
```
Provisioning Channel: Virtual Appliance (AD Connector)

Configuration:
  Channel: VA-Based
  VA Name: VA-PROD-01
  Protocol: LDAP + PowerShell
  Domain Controller: dc01.company.com
  
Provisioning Flow:
  
  1. ISC creates provisioning task
     Operation: Create AD Account
     User: john.smith
     
  2. ISC sends command to VA-PROD-01
     Command queued in VA
     
  3. VA receives command
     Provisioning task: Create account
     
  4. VA connects to DC (dc01.company.com)
     Protocol: LDAP (port 389 or 636)
     Authentication: Service account credentials
     
  5. VA executes PowerShell:
     New-ADUser -Name "john.smith" `
       -GivenName "John" `
       -Surname "Smith" `
       -SamAccountName "john.smith" `
       -UserPrincipalName "john.smith@company.com" `
       -EmailAddress "john.smith@company.com" `
       -Department "Sales" `
       -Path "OU=Sales,OU=Users,DC=company,DC=com" `
       -AccountPassword (ConvertTo-SecureString "TempP@ss123!" -AsPlainText -Force) `
       -Enabled $true
     
  6. AD creates account
     Account: john.smith created
     
  7. VA receives success from AD
     Account creation confirmed
     
  8. VA reports to ISC
     Status: Success
     Account: john.smith
     DN: CN=john.smith,OU=Sales,OU=Users,DC=company,DC=com
     
  9. ISC records successful provisioning
     Account linked to identity
     
Duration: 5-15 seconds (depends on network/VA load)
```

**Example 2: Oracle Database**
```
Provisioning Channel: Virtual Appliance (JDBC Connector)

Configuration:
  Channel: VA-Based
  VA Name: VA-PROD-01
  Protocol: JDBC
  Database Host: oracledb.company.com:1521
  Service Name: HRPROD
  
Provisioning Flow:
  
  1. ISC sends provisioning command to VA
     Operation: Create user in HR database
     
  2. VA connects to Oracle database
     JDBC Connection String:
       jdbc:oracle:thin:@oracledb.company.com:1521:HRPROD
     
  3. VA executes SQL:
     INSERT INTO users (
       user_id,
       username,
       first_name,
       last_name,
       email,
       department,
       created_date
     ) VALUES (
       SEQ_USER_ID.NEXTVAL,
       'john.smith',
       'John',
       'Smith',
       'john.smith@company.com',
       'Sales',
       SYSDATE
     );
     
     COMMIT;
     
  4. Oracle executes SQL
     Row inserted
     User_ID: 12345 (auto-generated)
     
  5. VA receives success
     1 row affected
     
  6. VA reports to ISC
     Status: Success
     User_ID: 12345
     
Duration: 3-10 seconds
```

## Channel 3: Web Services (REST/SOAP)

### **How It Works**

**Architecture:**
```
ISC → HTTP/HTTPS → Web Service API → Account Created

Similar to Direct API but more generic
Supports custom applications
Flexible integration
```

**Characteristics:**
```
✓ Flexible (works with custom apps)
✓ Standard protocols (REST/SOAP)
✓ Can be direct or via VA
✓ Supports JSON, XML, other formats
✓ Customizable

✗ Requires API development (if custom)
✗ May need custom configuration
✗ Varies by implementation
```

### **REST Example**
```
Provisioning Channel: Web Services (REST API)

Target: Custom HR Application

Configuration:
  Base URL: https://hr.company.com/api/v1
  Authentication: Bearer Token
  Format: JSON
  
Provisioning Flow:
  
  1. ISC calls REST endpoint:
     POST /api/v1/employees
     
     Headers:
       Authorization: Bearer {api_token}
       Content-Type: application/json
     
     Body:
       {
         "employeeId": "EMP12345",
         "firstName": "John",
         "lastName": "Smith",
         "email": "john.smith@company.com",
         "department": "Sales",
         "hireDate": "2024-02-14"
       }
  
  2. HR Application processes request
     Creates employee record
     
  3. HR Application responds:
     HTTP 201 Created
     {
       "id": 12345,
       "employeeId": "EMP12345",
       "status": "created"
     }
  
  4. ISC records success
     Employee created: 12345
```

## Channel 4: SCIM (System for Cross-domain Identity Management)

### **What is SCIM?**

**Definition:** Standard protocol for user provisioning across systems

**Purpose:**
```
Industry standard for provisioning
Consistent API across applications
Simplified integration
```

**SCIM Versions:**
```
SCIM 1.1: Older version
SCIM 2.0: Current standard (preferred)
```

### **How SCIM Works**

**Architecture:**
```
ISC (SCIM Client) → SCIM API → Application → Account Created

Standard operations:
  - POST /Users (create)
  - PUT /Users/{id} (update)
  - PATCH /Users/{id} (partial update)
  - DELETE /Users/{id} (delete)
  - GET /Users (list)
  - GET /Users/{id} (retrieve)
```

### **SCIM Example**
```
Provisioning Channel: SCIM 2.0

Target: Slack (supports SCIM 2.0)

Configuration:
  SCIM Endpoint: https://api.slack.com/scim/v2
  Authentication: Bearer Token
  Version: SCIM 2.0
  
Provisioning Flow:
  
  1. ISC creates user via SCIM:
     POST /scim/v2/Users
     
     Headers:
       Authorization: Bearer {scim_token}
       Content-Type: application/scim+json
     
     Body:
       {
         "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
         "userName": "john.smith@company.com",
         "name": {
           "givenName": "John",
           "familyName": "Smith"
         },
         "emails": [{
           "value": "john.smith@company.com",
           "primary": true
         }],
         "active": true
       }
  
  2. Slack processes SCIM request
     Creates user account
     
  3. Slack responds:
     HTTP 201 Created
     {
       "id": "U012AB3CD",
       "userName": "john.smith@company.com",
       "active": true
     }
  
  4. ISC records account created
     Slack User ID: U012AB3CD
```

**Applications supporting SCIM:**
```
Common SCIM-enabled applications:
  ✓ Slack
  ✓ Zoom
  ✓ GitHub
  ✓ Atlassian (Jira, Confluence)
  ✓ Dropbox Business
  ✓ Box
  ✓ Many others
  
Advantage:
  Standardized provisioning across all these apps
  Consistent implementation
```

## Channel 5: Cloud Gateway

### **What is Cloud Gateway?**

**Definition:** SailPoint-managed cloud service that connects to specific cloud providers

**Purpose:**
```
Streamlined integration with major cloud platforms
Managed by SailPoint
No VA required for cloud-to-cloud
```

### **Example: Azure AD**
```
Provisioning Channel: Cloud Gateway (Azure AD Connector)

Configuration:
  Channel: Cloud Gateway
  Provider: Microsoft Azure AD
  Authentication: Azure AD App Registration
  
Architecture:
  ISC → Cloud Gateway → Microsoft Graph API → Azure AD → Account Created
  
Advantages:
  ✓ No VA needed
  ✓ Optimized for Azure AD
  ✓ Automatic updates
  ✓ High performance
  ✓ Microsoft-certified integration
  
Provisioning Flow:
  1. ISC sends command to Cloud Gateway
  2. Cloud Gateway calls Microsoft Graph API
  3. Azure AD creates/updates user
  4. Cloud Gateway reports to ISC
```

## Channel Selection Summary

### **Decision Matrix**

| Source Type | Location | Recommended Channel | Why |
|-------------|----------|---------------------|-----|
| **Salesforce** | Cloud | Direct API | Cloud-to-cloud, native connector |
| **Active Directory** | On-premises | Virtual Appliance | On-prem, LDAP protocol |
| **Azure AD** | Cloud | Cloud Gateway | Optimized Azure connector |
| **Custom App (REST)** | Cloud | Direct API / Web Services | Standard REST API |
| **Custom App (REST)** | On-premises | VA + Web Services | On-prem REST API |
| **Oracle Database** | On-premises | Virtual Appliance (JDBC) | On-prem database |
| **Slack** | Cloud | Direct API (SCIM) | Cloud, supports SCIM 2.0 |
| **Legacy System** | On-premises | Virtual Appliance | Only accessible on-prem |

```
  Salesforce uses no VA (direct)
```

**Channel Selection Rationale:**
Location-Based Decision:
If Cloud → Direct API
- Fastest
- Simplest
- No VA needed
If On-Premises → Virtual Appliance
- Required for network access
- Secure architecture
- Supports multiple sources
If On-Premises + REST API → VA + Web Services
- Flexible
- Standardized
- Secure

**Exam Tip:** Cloud sources use Direct API, on-premises sources require Virtual Appliance. One VA can serve multiple on-premises sources (AD, databases, custom apps).

## Section Summary

**Core Concepts for Certification:**

✅ **Provisioning Channels:**
- **Direct API**: Cloud-to-cloud, no VA (Salesforce, ServiceNow)
- **Virtual Appliance**: On-premises access (Active Directory, databases)
- **Web Services**: REST/SOAP APIs (custom apps)
- **SCIM**: Standard provisioning protocol (Slack, Zoom)
- **Cloud Gateway**: SailPoint-managed cloud connectors (Azure AD)

✅ **Channel Selection:**
- **Cloud source** → Direct API (fastest, simplest)
- **On-premises source** → Virtual Appliance (required)
- **Custom on-prem API** → VA + Web Services
- **One VA** can serve multiple on-premises sources

✅ **Key Characteristics:**
- **Direct API**: Real-time, no VA, cloud-to-cloud
- **VA**: Secure (outbound only), reaches on-prem, extra hop
- **SCIM**: Standard protocol, consistent across apps
- **Web Services**: Flexible, customizable, REST/SOAP

✅ **Decision Factors:**
- Source location (cloud vs. on-premises)
- Protocol supported (API, LDAP, JDBC)
- Security requirements
- Performance needs
- Infrastructure availability

**Exam Focus Areas:**
- Know channel types and when to use each
- Understand Direct API for cloud sources
- Know VA required for on-premises sources
- Remember one VA can serve multiple sources
- Understand SCIM as standard provisioning protocol
