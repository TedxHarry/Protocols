# 5.2.3 Account Schemas and Attributes

**Learning Objective:** Understand account schemas, how they define provisioning structure, and attribute mapping.

---

## What is an Account Schema?

**Definition:** The structure that defines what attributes an account has in a source system

**Simple concept:**
```
Account Schema = Blueprint for accounts in a system

Defines:
  - What attributes exist (name, email, department)
  - What data types (string, integer, date)
  - Which are required vs. optional
  - Which are single-valued vs. multi-valued
```

**Example - Active Directory Schema:**
```
Attributes:
  sAMAccountName: String, Required, Single-valued
  givenName: String, Optional, Single-valued
  sn: String, Optional, Single-valued
  mail: String, Optional, Single-valued
  department: String, Optional, Single-valued
  memberOf: String, Optional, Multi-valued (list of groups)
```

**Exam Tip:** Schema defines the structure of accounts in a source. ISC must understand schema to provision correctly.

---

## How ISC Discovers Schemas

### **Schema Discovery Process**

**During source configuration:**
```
1. ISC connects to source
   Via API or VA

2. ISC queries source schema
   "What attributes exist in this system?"

3. Source returns schema
   List of all available attributes

4. ISC stores schema
   Used for provisioning and aggregation

5. Admin can customize
   Select which attributes to use
   Map ISC attributes to source attributes
```

**Example - Active Directory:**
```
ISC connects to AD via VA:
  
VA queries AD schema:
  Get-ADUser -Properties * (sample account)
  
Returns attributes:
  - sAMAccountName
  - displayName
  - givenName
  - sn (surname)
  - mail
  - department
  - title
  - manager
  - employeeID
  - memberOf
  - ... (100+ attributes)

ISC stores schema:
  Available AD attributes known
  Can be used for provisioning
```

## Schema Components

### **Attribute Properties**

**1. Attribute Name**
```
The identifier for the attribute

Examples:
  Active Directory: sAMAccountName, givenName, mail
  Salesforce: Username, Email, FirstName
  
Note: Names vary by source type
  ISC: email
  AD: mail
  Salesforce: Email
  
Mapping needed to connect them
```

---

**2. Data Type**

| Data Type | Description | Example Values |
|-----------|-------------|----------------|
| **String** | Text | "John Smith", "Sales" |
| **Integer** | Whole number | 12345, 100 |
| **Boolean** | True/False | true, false |
| **Date** | Date/DateTime | 2024-02-14, 2024-02-14T10:30:00Z |
| **Multi-valued String** | List of strings | ["VPN-Users", "Sales-Team"] |

**Example - Active Directory:**
```
givenName: String
  Value: "John"

employeeID: String (or Integer)
  Value: "12345"

accountExpires: Integer (timestamp)
  Value: 133515552000000000

memberOf: Multi-valued String
  Value: [
    "CN=VPN-Users,OU=Groups,DC=company,DC=com",
    "CN=Sales-Team,OU=Groups,DC=company,DC=com"
  ]

userAccountControl: Integer (flags)
  Value: 512 (normal account enabled)
```

---

**3. Required vs. Optional**

**Required attributes:**
```
Must be provided when creating account
Account creation fails if missing

Active Directory Required:
  - sAMAccountName (username)
  
Salesforce Required:
  - Username
  - Email
  - LastName
  - Alias
  - TimeZoneSidKey
  - LocaleSidKey
  - EmailEncodingKey
  - LanguageLocaleKey
  - ProfileId
```

**Optional attributes:**
```
Can be omitted when creating account
Defaults used if not provided

Active Directory Optional:
  - givenName (first name)
  - sn (last name)
  - mail (email)
  - department
  - title
  - manager
  
Can create account without these
But recommended to populate them
```

---

**4. Single-valued vs. Multi-valued**

**Single-valued:**
```
Can have only ONE value

Examples:
  - email: john.smith@company.com (one email)
  - department: Sales (one department)
  - manager: Jane Doe (one manager)
```

**Multi-valued:**
```
Can have MULTIPLE values

Examples:
  - memberOf: [VPN-Users, Sales-Team, All-Employees]
  - proxyAddresses: [smtp:john.smith@company.com, smtp:jsmith@company.com]
  
Stored as array/list
ISC can add/remove individual values
```

---

**5. Read-Only vs. Writable**

**Read-only attributes:**
```
ISC can read but cannot modify

Active Directory Examples:
  - whenCreated (creation timestamp)
  - whenChanged (last modified timestamp)
  - objectGUID (unique identifier)
  - objectSid (security identifier)

Why read-only:
  - System-managed
  - Automatically calculated
  - Security-sensitive
```

**Writable attributes:**
```
ISC can read and modify

Active Directory Examples:
  - givenName
  - sn
  - mail
  - department
  - title
  - manager
  - memberOf (via group operations)

ISC can provision these
```

## Attribute Mapping

### **What is Attribute Mapping?**

**Definition:** Connecting ISC identity attributes to source account attributes

**Purpose:**
```
ISC needs to know:
  ISC has: firstName
  AD expects: givenName
  
Mapping: firstName → givenName
  
When provisioning:
  ISC.firstName "John" → AD.givenName "John"
```

### **Mapping Configuration**

**Basic mapping:**
```
ISC Attribute    →    Source Attribute

firstName        →    givenName
lastName         →    sn
email            →    mail
employeeNumber   →    employeeID
department       →    department
```

**Example provisioning with mapping:**
```
Identity in ISC:
  firstName: "John"
  lastName: "Smith"
  email: "john.smith@company.com"
  employeeNumber: "EMP12345"
  department: "Sales"

Mapping applied:
  givenName = "John"         (from firstName)
  sn = "Smith"               (from lastName)
  mail = "john.smith@company.com"  (from email)
  employeeID = "EMP12345"    (from employeeNumber)
  department = "Sales"       (from department)

AD account created with:
  sAMAccountName: john.smith
  givenName: John
  sn: Smith
  mail: john.smith@company.com
  employeeID: EMP12345
  department: Sales
```

---

### **Complex Mappings**

**Calculated attributes:**
```
Mapping with transform:

ISC: firstName, lastName
AD: displayName

Mapping: displayName = firstName + " " + lastName

Result:
  firstName: "John"
  lastName: "Smith"
  displayName: "John Smith" (calculated)
```

**Conditional mappings:**
```
Mapping with rule:

ISC: department
AD: OU (organizational unit)

Mapping Rule:
  IF department = "Sales" → OU = "OU=Sales,OU=Users,DC=company,DC=com"
  IF department = "IT" → OU = "OU=IT,OU=Users,DC=company,DC=com"
  ELSE → OU = "OU=Users,DC=company,DC=com"

Result:
  department: "Sales"
  OU: "OU=Sales,OU=Users,DC=company,DC=com"
```

**Static values:**
```
Mapping to constant:

AD: company
Mapping: company = "Acme Corporation" (static)

Result:
  All accounts get: company = "Acme Corporation"
```

### **Account Attributes (Individual)**

**User information:**
```
Category: Personal/Contact

ISC → Source Mappings:
  firstName → givenName
  lastName → sn
  displayName → displayName
  email → mail
  phone → telephoneNumber
  mobile → mobile

Example:
  givenName: "John"
  sn: "Smith"
  displayName: "John Smith"
  mail: "john.smith@company.com"
  telephoneNumber: "+1-555-0100"
```

**Organizational information:**
```
Category: Organization

ISC → Source Mappings:
  department → department
  title → title
  manager → manager
  location → physicalDeliveryOfficeName
  costCenter → costCenter
  employeeNumber → employeeID

Example:
  department: "Sales"
  title: "Sales Representative"
  manager: "CN=Jane Doe,OU=Users,DC=company,DC=com"
  employeeID: "EMP12345"
```

**Account control:**
```
Category: Account Settings

Attributes:
  - accountExpires (expiration date)
  - userAccountControl (enabled/disabled, flags)
  - passwordNeverExpires (boolean)
  - mustChangePassword (boolean)

Example:
  userAccountControl: 512 (enabled, normal account)
  passwordNeverExpires: false
  mustChangePassword: true
```

**Group memberships:**
```
Attribute: memberOf (AD), groupMembership (generic)

  - CN=Sales-Team,OU=Groups,DC=company,DC=com
  - CN=All-Employees,OU=Groups,DC=company,DC=com

Provisioning operations:
  - Add to group (append to list)
  - Remove from group (remove from list)
```

**Role assignments:**
```
Salesforce Example:

Attribute: ProfileId, UserRoleId

Values:
  ProfileId: "00exx000001abcd" (Standard User)
  UserRoleId: "00Exx000000efgh" (Sales Rep)

Provisioning:
  ISC maps role to ProfileId
  Sets on account creation/update
```

## Schema Refresh

### **When to Refresh Schema**

**Triggers for refresh:**
```
✓ Source schema changes
  - New attributes added
  - Attributes removed
  - Data types changed
  
✓ New connector version
  - Updated connector may expose new attributes
  
✓ Customizations made
  - Extended schema
  - Added custom attributes
  
✓ Troubleshooting
  - Verify schema current
  - Check for inconsistencies
```

### **How to Refresh**

**In ISC:**
```
1. Navigate to source
   Admin → Connections → Sources → [Source]

2. Click "Refresh Schema" or "Re-discover"

3. ISC queries source
   Retrieves current schema
   
4. ISC updates schema
   New attributes available
   Old attributes removed if gone
   
5. Review changes
   Check new attributes
   Update mappings if needed
   
6. Test provisioning
   Verify still works correctly
```

## Common Schema Issues

### **Issue 1: Required Attribute Missing**

**Problem:**
```
Account creation fails
Error: "Required attribute missing"

Example:
  Creating Salesforce account
  Error: "Required field missing: Alias"
```

**Solution:**
```
1. Identify required attribute
   Check schema: Alias is required
   
2. Add to mapping
   ISC.uid → Salesforce.Alias
   
3. Ensure ISC has value
   uid must be populated in identity
   
4. Retry provisioning
   Should succeed with all required fields
```

---

### **Issue 2: Data Type Mismatch**

**Problem:**
```
Attribute update fails
Error: "Invalid data type"

Example:
  Provisioning employeeID
  Source expects: Integer
  ISC sends: String "EMP12345"
  Error: Cannot convert to integer
```

**Solution:**
```
1. Check source schema
   employeeID: Integer type
   
2. Check ISC data
   employeeNumber: "EMP12345" (string with letters)
   
3. Fix mapping
   Option A: Change source to accept string
   Option B: Use numeric-only employeeId
   Option C: Transform (remove "EMP" prefix)
   
4. Update and retry
```

---

### **Issue 3: Attribute Not Writable**

**Problem:**
```
Update fails
Error: "Cannot modify read-only attribute"

Example:
  Trying to set whenCreated
  Error: whenCreated is read-only
```

**Solution:**
```
1. Check schema
   whenCreated: Read-only ✓
   
2. Remove from mapping
   Don't provision read-only attributes
   
3. Use writable alternative
   If tracking creation, use different attribute
```

**ACCOUNT CONTROL ATTRIBUTES**

**13. Account Expiration**
```
ISC Attribute: endDate (if applicable)
AD Attribute: accountExpires
Type: Integer (timestamp)
Required: No

Mapping:
  IF ISC.endDate exists:
    AD.accountExpires = Convert to AD timestamp
  ELSE:

  ISC.endDate = "2024-12-31"
  AD.accountExpires = 133515552000000000

  ✓ Contractor expiration
  ✓ Temporary accounts
  ✓ Automatic disabling on date
```

**14. User Account Control**
```
ISC Attribute: cloudLifecycleState
AD Attribute: userAccountControl
Type: Integer (flags)
Required: No (has default)

Mapping Logic:
  IF cloudLifecycleState = "active":
    AD.userAccountControl = 512 (enabled, normal account)
  ELSE IF cloudLifecycleState = "inactive":
    AD.userAccountControl = 514 (disabled)
  ELSE IF cloudLifecycleState = "leave":
    AD.userAccountControl = 514 (disabled)

Why Critical:
  ✓ Enable/disable accounts
  ✓ Account flags
  ✓ Security
```

## Section Summary

**Core Concepts for Certification:**

✅ **Provisioning Policies:**
- Configure HOW provisioning happens for each source
- Settings: Enable/disable, allowed operations, timing, method
- Operations: Create, Update, Delete, Enable, Disable
- Timing: Real-time (urgent) vs. Scheduled (batch)
- Best practice: Disable real-time, Delete scheduled (retention)

✅ **Provisioning Operations:**
- **Create**: Make new account (new hire, access request)
- **Update**: Modify existing (attribute/group changes)
- **Delete**: Remove account (after retention period)
- **Enable/Disable**: Activate/deactivate accounts

✅ **Account Schemas:**
- Define structure of accounts in source
- Specify attributes, data types, required/optional
- ISC discovers schema from source
- Used for provisioning and aggregation

✅ **Attribute Mapping:**
- Connect ISC attributes to source attributes
- ISC.firstName → AD.givenName
- ISC.email → AD.mail
- Essential for provisioning to work correctly

✅ **Key Attributes to Map:**
- **Required**: sAMAccountName (username)
